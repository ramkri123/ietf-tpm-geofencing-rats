Slide 1: Why Workload Geofencing Needs Hardware Roots

  • Token theft & replay — Bearer tokens and software-only identities can be stolen and used from any location
  • Spoofable location — IP geolocation, VPNs, cloud region labels provide zero cryptographic assurance
  • Regulatory pressure — RBI India, South Korea Spatial Data Act require provable data residency
  • Unsigned location metadata — No integrity, no audit trail, no verifiable provenance
  • The gap — No cryptographic link between a workload's identity, its platform integrity, and its physical location
Speaker note: "The core gap is: today, nothing binds WHERE a workload runs to WHO it claims to be, with hardware-rooted proof."

Slide 2: V-GAP — Verifiable Geofencing Attestation Profile

  • RATS Architecture profile (RFC 9334) — builds on existing RATS plumbing
  • Binds workload identity issuance to two hardware-rooted proofs:
    • Platform integrity — TPM quote seals all evidence fields
    • Geographic residency — "in-zone" proof, optionally privacy-preserving via transparent ZKPs
  • Credential issued only when both pass — fail-closed via X.509 CRITICAL extension

Three-Layer Trust Chain (silicon → agent → workload)

LayerWhatHow
Layer 1Workload ↔ Agent bindingTransitive Attestation (draft-mw-wimse)
Layer 2Platform integrityTPM quote + PCR validation (this document)
Layer 3Residency verificationGeolocation proof + MNO endorsement (this document)
Speaker note: "V-GAP is Layer 2 + Layer 3. Layer 1 is handled by transitive attestation. Together they give you silicon-to-workload trust."

Evidence Flow (background-check model, RFC 9334 §3.2)

LAH (Attester) ──→ lah-bundle ──→ Verifier ◄── MNO Endorsement
                                      │
                              Attestation Result
                                      │
                                      ▼
                      RP + CA ──→ X.509-SVID (CRITICAL ext)
                                      │
                                      ▼
                             Downstream RP (mTLS)

Key design: RP acts as "trust translator" — embeds attestation result into standard X.509, so downstream consumers don't need to understand RATS or V-GAP.

Speaker note: "This is the background-check model from RFC 9334. The Attester never sees the result — it goes straight from Verifier to RP. The RP bakes it into a standard cert."
